# webserv 과제란 무엇인가

webserv 과제는
“웹 서버를 하나 만들어 보라”는 과제처럼 보이지만,
실제로는 **서버 프로그램의 본질적인 구조를 이해했는지**를 묻는 과제이다.

이 과제의 목표는
완성도 높은 웹 서버를 만드는 것이 아니라,
웹 서버가 어떤 문제를 해결하기 위해
어떤 구조를 가질 수밖에 없는지를
직접 구현을 통해 경험하는 데 있다.

---

## webserv는 웹 프레임워크가 아니다
webserv는
Spring, Django, Express 같은
웹 프레임워크를 만드는 과제가 아니다.

데이터베이스도 없고,
사용자 상태를 저장하지도 않으며,
복잡한 비즈니스 로직을 요구하지도 않는다.

대신 webserv는
**가장 단순한 웹 서버의 형태**를 구현하게 만든다.
요청을 받고,
규칙에 맞게 해석하고,
적절한 응답을 돌려주는 것.
이 단순한 흐름이 과제의 핵심이다.

---

## 이 과제가 다루는 핵심 질문
webserv 과제를 관통하는 질문은 다음과 같다.

- 하나의 서버는 어떻게 여러 클라이언트를 동시에 상대할 수 있는가?
- 네트워크 요청은 언제, 어떻게 처리되어야 하는가?
- 서버는 언제 기다려야 하고, 언제 반응해야 하는가?
- 외부 프로그램(CGI)은 서버 구조 안에서 어떻게 다뤄져야 하는가?

이 질문들에 대한 답을
**코드로 직접 증명**하는 것이
webserv 과제이다.

---

## 왜 이벤트 기반 서버인가
webserv는
단순한 순차 처리 서버를 허용하지 않는다.
여러 클라이언트의 요청을
하나씩 blocking 방식으로 처리한다면,
서버는 금방 멈춰 버릴 것이다.

그래서 이 과제는
이벤트 기반 서버 구조를 요구한다.
이벤트 루프,
non-blocking I/O,
그리고 poll과 같은 메커니즘은
선택이 아니라 필연이다.

이 구조를 이해하지 못하면
과제의 요구사항은
겉보기에는 구현할 수 있어 보이지만,
조금만 복잡해져도
안정성을 유지할 수 없다.

---

## webserv에서 CGI가 등장하는 이유
정적 웹 서버만으로는
모든 요청을 처리할 수 없다.
간단한 계산,
동적인 응답,
또는 외부 프로그램의 실행이 필요한 경우가 발생한다.

CGI는
이 문제를 해결하기 위한
가장 단순한 확장 방식이다.
서버는 자신의 역할을 유지한 채,
필요한 경우에만
외부 프로그램에게 처리를 위임한다.

CGI를 통해
서버는 “무엇을 직접 처리하고,
무엇을 넘겨야 하는가”라는
책임 분리를 배우게 된다.

---

## 이 과제가 진짜로 요구하는 것
webserv는
완벽한 HTTP 서버를 요구하지 않는다.
모든 RFC를 구현할 필요도 없고,
실제 서비스 수준의 성능도 기대하지 않는다.

대신 이 과제는
다음과 같은 능력을 요구한다.

- 서버 프로그램을 구조적으로 설계하는 능력
- 네트워크 I/O의 특성을 이해하는 능력
- 상태를 관리하며 이벤트를 처리하는 사고 방식
- 예외 상황에서도 서버를 안정적으로 유지하는 판단

이 능력들은
특정 언어나 기술에 종속되지 않는다.
webserv는
서버를 만드는 사고방식을
훈련시키는 과제이다.

---

## 무엇을 구현해야 하는가
webserv 과제는
“무엇이든 되는 서버”를 요구하지 않는다.
대신,
명확하게 정의된 최소 기능을
안정적으로 구현할 것을 요구한다.

구현해야 하는 대상은 다음과 같다.

- 클라이언트의 연결을 수락하고 관리하는 서버
- HTTP 요청을 규격에 맞게 읽고 해석하는 로직
- 요청에 따라 정적 파일을 제공하는 기능
- 업로드와 같은 간단한 파일 처리
- 오류 상황에 맞는 HTTP 응답 생성
- 여러 클라이언트를 동시에 처리할 수 있는 구조
- CGI를 통한 외부 프로그램 실행

중요한 점은,
각 기능 자체보다도
**이 기능들을 어떤 구조로 연결했는가**이다.

webserv는
기능을 많이 구현하는 과제가 아니라,
기능들이 충돌하지 않도록
구조를 설계하는 과제이다.

---

## 무엇을 공부해야 하는가
webserv를 완주하기 위해 필요한 것은
특정 함수 목록이나
트릭성 구현 방법이 아니다.

이 과제를 통해 반드시 마주치게 되는 학습 주제는 다음과 같다.

- 서버는 왜 이벤트 기반 구조를 가질 수밖에 없는가
- blocking I/O가 서버 전체에 어떤 영향을 주는가
- 파일 디스크립터를 상태로 관리한다는 의미
- 요청과 응답을 시간의 흐름으로 바라보는 사고
- CGI와 같은 외부 프로세스를 서버 구조 안에서 다루는 방법
- timeout, abort와 같은 예외 상황에서의 판단 기준

이 개념들은
webserv 과제에만 국한되지 않는다.
네트워크 서버,
게임 서버,
백엔드 시스템 전반에 공통으로 등장하는 문제들이다.

따라서 webserv는
특정 기술을 익히는 과제가 아니라,
**서버를 바라보는 사고방식을 학습하는 과제**에 가깝다.

---

## 무엇을 알고 있어야 하는가
webserv 과제를 진행하기 위해
모든 개념을 처음부터 완벽히 알고 있을 필요는 없다.
하지만 과제를 진행하면서
반드시 마주치게 되는 개념들이 존재한다.

다음은 webserv를 이해하고 설명하기 위해
알고 있어야 하는 핵심 개념들이다.

- **웹과 클라이언트–서버 모델**
- **TCP 연결과 소켓**
- **파일 디스크립터(FD)**
- **blocking I/O와 non-blocking I/O**
- **이벤트 루프(Event Loop)**
- **poll과 같은 이벤트 감시 메커니즘**
- **HTTP 요청과 응답의 구조**
- **상태 관리와 상태 기계(State Machine)**
- **CGI와 외부 프로세스**
- **timeout과 abort 상황**

이 개념들은
암기 대상이 아니라
과제를 진행하며
자연스럽게 체득하게 되는 것들이다.

webserv는
이 개념들을 **알고 있는지**를 묻기보다,
이 개념들을 **이해한 상태로 설명할 수 있는지**를 묻는 과제이다.

---

## 이 문서와 webserv 과제의 관계
이 문서는
webserv 과제를 진행하며
왜 이런 구조를 선택했는지,
왜 이런 구현이 필요했는지를
정리한 기록이다.

정답 코드를 제공하지도 않고,
가장 빠른 구현 방법을 알려주지도 않는다.
대신,
**설명할 수 있는 서버를 만들기까지의 과정**을 담고 있다.

이 문서를 통해
webserv 과제가 끝난 이후에도,
서버라는 개념을
다시 돌아보고 설명할 수 있기를 바란다.
