Server_Manager 아키텍처

이 문서는 webserv 프로젝트에서 Server_Manager가 왜 존재하며,
무엇을 책임지고 무엇을 절대 하지 않는지,
그리고 실제 코드 기준으로 어떻게 동작하는지를 설명한다.

Server_Manager는 HTTP 서버가 아니다.
Server_Manager는 상태 머신도 아니다.

이 객체의 역할은 단 하나다.

“파일 디스크립터 이벤트를 관제하고,
적절한 객체(Client)에게 ‘지금 이걸 해라’라고 지시하는 것”

1. Server_Manager의 존재 이유​

1.1 왜 Server_Manager가 필요한가​

webserv는 다음과 같은 요구를 동시에 만족해야 한다.

  • 여러 listening socket 관리
  • 수십~수백 개의 client socket 동시 처리
  • CGI 프로세스의 stdin / stdout 파이프 비동기 처리
  • idle timeout 기반 연결 정리
  • 모든 FD를 단일 이벤트 루프에서 poll 기반으로 처리

이 모든 FD를 각 객체(Client, CGI, Server)가 직접 poll에서 관리하게 만들면 구조는 즉시 붕괴한다.

따라서 webserv는 다음 구조를 선택했다.

  • Server_Manager
    • poll 루프를 소유
    • 모든 FD 이벤트를 수집
    • FD의 “종류”에 따라 처리 흐름을 분기
  • Client
    • 요청 파싱, 상태 머신, 응답 생성, CGI 실행 등 “의미 있는 로직” 담당

Server_Manager는 지휘자,
Client는 연주자다.

1.2 Server_Manager가 하지 않는 것​

Server_Manager는 다음을 절대 하지 않는다.

  • HTTP 요청을 파싱하지 않는다
  • HTTP 상태 코드를 판단하지 않는다
  • keep-alive 여부를 직접 결정하지 않는다
  • CGI 로직을 해석하지 않는다
  • 파일 시스템에 접근하지 않는다

이 모든 책임은 Client, RequestParser, Response_Builder에 있다.

Server_Manager는 의미를 해석하지 않고, 타이밍만 통제한다.

2. Server_Manager의 책임 범위 (코드 기준)​

Server_Manager가 실제 코드에서 책임지는 영역은 다음뿐이다.

2.1 이벤트 루프 소유​

  • run() 함수가 poll 기반 메인 루프를 소유
  • poll(_poll_fds, TIMEOUT_MS) 호출
  • SIGINT 시 g_running 플래그로 안전 종료
while (g_running)
{
    poll(...)
    ...
}
2.2 FD 생명주기 관리
Server_Manager는 FD의 생성·등록·제거를 모두 통제한다.

listening socket FD

client socket FD

CGI stdin FD

CGI stdout FD

모든 FD는 _poll_fds에 등록되지 않으면 존재 의미가 없다.

2.3 Client 객체 생성 및 파괴
accept_new_client()에서 Client 생성

_clients[fd] = Client(...) 형태로 관리

close_connection()에서 Client 제거 + FD 정리

Client는 항상 Server_Manager에 의해 태어나고, Server_Manager에 의해 죽는다.

2.4 CGI 파이프 FD 관리
CGI는 일반 소켓과 완전히 다르다.

client socket → 양방향 하나

CGI → stdin FD / stdout FD 두 개

Server_Manager는 이를 다음 구조로 관리한다.

cpp
코드 복사
std::map<int, CgiFdInfo> _cgi_fd_map;
key: pipe fd

value: 어떤 Client의 FD인지 + stdin/stdout 여부

2.5 idle timeout 관리
_tick_counter는 /proc/uptime 기반 ms 타이머

Client의 last_activity_tick 기준으로 idle 판단

CGI 실행 중이거나 응답 전송 중이면 idle 검사 제외

cpp
코드 복사
if (now - last >= IDLE_TIMEOUT_MS_VALUE)
    close_connection(fd);
3. Server_Manager가 소유하는 핵심 데이터 구조
3.1 _poll_fds
cpp
코드 복사
std::vector<struct pollfd> _poll_fds;
webserv 전체에서 유일한 poll 대상 컬렉션

FD 추가 / 제거는 반드시 Server_Manager를 통해서만 가능

Client는 pollfd를 직접 만지지 않는다

3.2 _clients
cpp
코드 복사
std::map<int, Client> _clients;
key: client socket FD

value: 해당 연결의 Client 객체

Server_Manager는 Client의 내부 상태를 해석하지 않는다

단지 get_state()로 상태 분기만 수행

3.3 _cgi_fd_map
cpp
코드 복사
std::map<int, CgiFdInfo> _cgi_fd_map;
CGI 파이프 FD → 어떤 Client의 어떤 스트림인지 매핑

CGI FD는 절대 Client FD와 같은 방식으로 처리되지 않는다

4. 초기화 단계 흐름
4.1 Server 정보 수신
생성자에서 std::vector<Server>를 전달받음

Server에는 이미 ConfigParser 결과가 들어 있음

4.2 listening socket 생성
init_sockets()에서 수행:

중복 listen target 방지 (host:port)

socket → setsockopt → bind → listen

non-blocking 설정

_listening_fds, _fd_to_server 등록

pollfd에 POLLIN으로 추가

5. 메인 이벤트 루프 구조
5.1 run() 전체 흐름 요약
text
코드 복사
poll
 ├─ CGI FD면 → handle_cgi_poll_event
 ├─ listening FD면 → accept_new_client
 ├─ client FD면
 │   ├─ POLLIN → receive_request
 │   ├─ 상태 검사 → build_response / CGI 등록
 │   └─ POLLOUT → send_response
 └─ 주기적으로 idle 검사
Server_Manager는 항상 FD의 종류부터 판별한다.

6. FD 타입별 처리 전략
6.1 listening FD
오직 accept_new_client()만 호출

요청 파싱, 상태 판단 없음

6.2 client socket FD
POLLIN
receive_request(fd)

내부에서 Client::handle_recv_data 호출

파싱 완료 시 Client::REQUEST_COMPLETE로 상태 변경

POLLOUT
send_response(fd)

부분 전송 지원

전송 완료 후 keep-alive 판단은 Client의 요청 정보 기반

6.3 CGI 파이프 FD
stdin FD → POLLOUT

stdout FD → POLLIN / POLLHUP

이벤트 발생 시 Client::handle_cgi_* 호출

CGI 종료 시 Client::handle_cgi_completion()

7. 요청 수신 흐름 (Server_Manager 관점)
POLLIN 발생

receive_request

Client가 내부 파싱

Server_Manager는 파싱 결과만 상태로 확인

REQUEST_COMPLETE 상태면 build_response 트리거

Server_Manager는 요청 내용을 절대 들여다보지 않는다.

8. 응답 전송 흐름 (Server_Manager 관점)
Client 상태가 SENDING_RESPONSE

pollfd를 POLLOUT으로 전환

send_response 호출

부분 전송 반복

전송 완료 후

keep-alive → reset

아니면 → close

9. CGI 처리 흐름 요약
CGI 시작은 Client가 요청

FD 등록/해제는 Server_Manager가 수행

stdin/out 이벤트는 poll에서 감시

종료 시 다시 client socket POLLOUT으로 복귀

10. Server_Manager의 설계 원칙 (코드로 증명됨)
상태를 결정하지 않는다

의미를 해석하지 않는다

오직 “언제 호출할지”만 책임진다

Server_Manager는 뇌가 아니다.
Server_Manager는 신경계다.
